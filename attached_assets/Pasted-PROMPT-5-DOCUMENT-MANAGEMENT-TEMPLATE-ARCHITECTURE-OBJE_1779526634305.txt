PROMPT 5 — DOCUMENT MANAGEMENT & TEMPLATE ARCHITECTURE

OBJECTIVE

Implement the central document and template architecture that will support all current and future legal documents, agreements, ownership records, declarations, notices, succession records, transfer documents, and governance documents.

This prompt establishes the foundation only.

Do not implement ownership calculations.
Do not implement contribution calculations.
Do not implement maturity workflows.
Do not implement agreement business logic.

Only implement the document architecture and management layer.

====================================================
CORE PRINCIPLES
====================================================

1. No template upload during project creation.

Project creation must never ask users to upload templates.

Templates are managed centrally.

2. Templates are system-managed assets.

Users select templates.
Users do not upload templates during operational workflows.

3. Person Registry remains the source of person information.

4. Project remains the source of project information.

5. Document generation consumes data from authoritative sources.

No duplicate storage of person details.

No duplicate storage of project details.

====================================================
DOCUMENT VARIABLE REGISTRY
====================================================

Implement central Document Variable Registry.

This is an internal mapping layer.

Not a separate user workflow.

Every placeholder used by any template must map to exactly one source.

Sources allowed:

Project Field

Person Registry Field

Schedule A Field

Calculated Field

System Generated Field

Examples:

{{PROJECT_NAME}}

{{LANDOWNER_NAME}}

{{DEVELOPER_NAME}}

{{PROJECT_CODE}}

{{AGREEMENT_DATE}}

{{CURRENT_DATE}}

{{WITNESS_1_NAME}}

{{PARTNER_ADDRESS}}

{{PROJECT_LOCATION}}

Do not hardcode replacements inside templates.

All variables must be registered centrally.

====================================================
VARIABLE MAPPING UI
====================================================

Implement Variable Mapping screen under:

Administration
  → Document Templates
     → Variable Mapping

Purpose:

Map template placeholders to registry variables.

Features:

List placeholders detected inside uploaded template

Show mapped source

Show status:

Mapped

Missing

Invalid

Unused

Validation before activation.

Template cannot become Active if required variables remain unresolved.

====================================================
TEMPLATE UPLOAD
====================================================

Allow upload of DOCX templates.

Store template file in object storage.

Parse placeholders.

Register placeholders automatically.

Display detected variables.

Allow mapping review.

Allow activation after validation.

Create Document Generation History.

Never overwrite generated documents.

====================================================
DOCUMENT VERSIONING
====================================================

Versioning required.

Statuses:

Draft

Active

Superseded

Archived

Only one active version per document type.

Older versions remain available.

Generation history must preserve original version used.

====================================================
DOCUMENT STORAGE
====================================================

Generated documents stored in object storage.

Database stores metadata only.

No local filesystem storage.

Support:

DOCX

PDF metadata support

Future formats

====================================================
VALIDATION RULES
====================================================

Before activation:

Template file exists

Variables parsed

Variable mappings validated

Required variables resolved

Version uniqueness validated

Before generation:

Template active

Variable mappings valid

Required data available

====================================================
PERSON REGISTRY INTEGRATION
====================================================

Document generation must consume person data directly from Person Registry.

No duplicate person fields.

Witness selection must use Person Registry records.

Nominees must use Person Registry records.

Claimants must use Person Registry records.

====================================================
PROJECT INTEGRATION
====================================================

Document generation must consume project information directly from Project records.

No duplicate project data.

====================================================

====================================================

Track:

Template upload

Template activation

Template supersession

Variable mapping changes

Document generation

Document download

Document archival

All actions must be auditable.
