STAGE 1 — MULTI-STORE INVENTORY FOUNDATION

Build a new module called Multi-Store Inventory & Stock Transfer System.

This must integrate with the existing ERP architecture:

- authentication system
- role system
- project access system
- lifecycle system
- inventory system
- production/store entry system

Do NOT rebuild existing modules.

====================================================
CORE BUSINESS RULE
====================================================

Physical stock movement must NEVER change:

- project ownership
- landowner rights
- developer rights
- profit rights
- Lifecycle Contribution Accounting (LCA)

Stock transfer changes ONLY physical location.

====================================================
CURRENT SITUATION
====================================================

Currently system supports only project-specific stores.

Now extend system to support:

1. Project Stores
2. Central Shared Stores
3. Overflow Stores

Central stores may contain stock from multiple projects simultaneously.

====================================================
CRITICAL ARCHITECTURE RULE
====================================================

Separate system into 3 independent layers:

1. Ownership Ledger
2. Inventory Ledger
3. Physical Location Ledger

Never combine them.

====================================================
BUYER DISPATCH INTEGRATION
====================================================

Buyer should NEVER manually choose store.

When buyer order approved:

Developer selects dispatch source.

Example:

Available:

Project Store A = 200kg
Central Store = 600kg

Developer selects source.

Generate pickup memo:

- buyer
- project
- store
- zone
- rack
- quantity
- memo code
- pickup status

Buyer only sees designated pickup location.

====================================================
PARTIAL DISPATCH SUPPORT
====================================================

Support partial dispatch.

Example:

Order:
1000kg

First pickup:
300kg

Remaining:
700kg

Status:
Partially dispatched

Track:

- ordered quantity
- dispatched quantity
- remaining quantity

Prevent dispatch exceeding remaining balance.

====================================================
RESERVED INVENTORY RULES
====================================================

Inventory states:

- available
- reserved
- dispatched
- sold

Formula:

Available =
Total − Reserved − Sold

Reserved inventory must be protected.

====================================================
TRANSFER + RESERVATION RULE
====================================================

Prevent accidental dispatch problems.

If inventory reserved for buyer:

Either:

1. prevent transfer

OR

2. automatically update dispatch source

Do not allow pickup memo to point to wrong store.

====================================================
PROJECT LIFECYCLE RESTRICTIONS
====================================================

Integrate with existing lifecycle system.

Certain project statuses may restrict:

- transfers
- sales
- dispatch
- reservations

Example:

Closed project should not allow active sales.

Use permission matrix based on project lifecycle status.

====================================================
AUDIT REQUIREMENTS
====================================================

Every movement must create audit records.

Track complete chain:

Collection
Store entry
Transfer
Reservation
Dispatch
Sale

Never allow silent inventory movement.

UI REQUIREMENTS

Design scalable architecture for future multi-store expansion.
