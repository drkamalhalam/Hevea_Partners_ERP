STAGE — RUBBER SALES, PAYMENT & DISPATCH MANAGEMENT SYSTEM

Build a complete Rubber Sales, Payment, Reservation, Invoice & Dispatch Management module for the plantation ERP.

This module must integrate with the already existing:

- authentication system
- role system
- project-wise access system
- lifecycle system
- inventory/store system
- multi-store transfer system
- dashboard architecture
- audit system

Do NOT rebuild existing modules.

====================================================
CORE BUSINESS PRINCIPLES
====================================================

Separate these concepts completely:

1. Sale Performer
2. Payment Receiver
3. Inventory Ownership
4. Physical Inventory Location
5. Money Custody Holder

These may all be different people.

Example:

Employee Ravi performs sale.

Buyer pays Developer account online.

Inventory belongs to Project A.

Rubber dispatched from Central Store.

System must support this separation cleanly.

====================================================
PAYMENT WORKFLOW
====================================================

Workflow:

1. Seller enters:
- quantity
- rate

2. System calculates amount automatically.

3. Seller clicks:
Receive Payment

4. Generate:
- payment request
- QR code
- order reference

5. Buyer pays.

6. Payment callback/webhook may report payment detected.

7. IMPORTANT:
System should NOT auto-complete transaction.

Instead:

Payment Status:
awaiting_manual_confirmation

8. Authorized user manually clicks:
Confirm Payment

Only after confirmation:

- invoice finalized
- inventory reduced
- money custody ledger updated
- dispatch enabled
- audit log generated

====================================================

====================================================
PAYMENT SAFETY RULES
====================================================

====================================================
INVENTORY RESERVATION ENGINE
====================================================

When payment request created:

Reserve inventory temporarily.

Inventory states:

- available
- reserved
- dispatched
- sold

Formula:

Available =
Total − Reserved − Sold

====================================================
PAYMENT EXPIRY LOGIC
====================================================

Payment requests may expire.

Example:

30 minute expiry.

- release reserved inventory
- mark request expired

====================================================
INCONSISTENCY PREVENTION
====================================================

Solve these issues carefully:

1. Duplicate payment clicks
2. Double inventory deduction
3. Payment detected but not confirmed
4. Reservation without payment
5. Dispatch exceeding remaining quantity
6. Selling already reserved stock
7. Wrong pickup location after transfer
8. Negative inventory
9. Unauthorized payment receiver
10. Unauthorized seller
11. Manual status manipulation
12. Expired unpaid reservations
13. Cash custody mismatch
14. Inventory totals not reconciling
15. Project lifecycle conflicts
